# Design Folder Organization

The goal is simple: keep capture effortless, keep current work visible, and stop forcing every document into a type-based taxonomy.

---

<!-- layout: split:text-text -->
## Why the Old Structure Creates Friction

The problem is not a lack of folders. It is too much precision too early.

<div>
  <h3>Mixed dimensions</h3>
  <p>Folders combine content type, life area, and intent, so every filing decision becomes interpretive work.</p>
  <h3>Micro-decisions everywhere</h3>
  <p>Notes versus snippets versus reference sounds tidy, but it creates hesitation at the moment of capture.</p>
</div>

<div>
  <h3>Knowledge turns into a junk drawer</h3>
  <p>When reusable material has one vague home, retrieval gets worse instead of better.</p>
  <h3>Inbox already proves the point</h3>
  <p>The current inbox works because it is low-friction, fast, and decision-light.</p>
</div>

## Rule Changes That Make It Work

The folder names matter, but the operating rules matter more.

<div>
  <h3>Default destination is <code>active/</code></h3>
  <p>When you are unsure, put it with current thinking instead of chasing a perfect category.</p>
  <h3>Stop distinguishing notes, ideas, and snippets</h3>
  <p>They are all documents. Let naming and search carry the meaning.</p>
</div>

<div>
  <h3>Flatten the tree</h3>
  <p>The guidance is to reduce depth aggressively and keep the structure close to two levels when possible.</p>
  <h3>Projects contain everything</h3>
  <p>No notes folder, ideas folder, or drafts folder inside the project. If it belongs, it stays there.</p>
</div>

## Personal vs Work Naming

The role is similar in both systems, but the document intentionally uses different names.

<div class="dz-card-grid" data-dz-columns="2">
  <div class="dz-card dz-tone-info fragment" data-fragment-index="1">
    <h3>Personal uses <code>library/</code></h3>
    <p>Broader, more idea-heavy, and better suited to a creative personal system.</p>
    <small>The personal tree emphasizes creativity, expression, and low-friction exploration.</small>
  </div>
  <div class="dz-card dz-tone-success fragment" data-fragment-index="2">
    <h3>Work uses <code>reference/</code></h3>
    <p>Practical, reusable, and easier to scan in a work context.</p>
    <small>The document explicitly says this rename matches how work brains think.</small>
  </div>
</div>

## Weekly Inbox Reset

The maintenance ritual is intentionally tiny so the system can survive real life.

<div class="dz-card-grid" data-dz-columns="4">
  <div class="dz-card dz-tone-info fragment" data-fragment-index="1">
    <strong>Open inbox</strong>
    <small>Start from the capture pile.</small>
  </div>
  <div class="dz-card dz-tone-success fragment" data-fragment-index="2">
    <strong>Still active?</strong>
    <small>Move it to <code>active/</code>.</small>
  </div>
  <div class="dz-card fragment" data-fragment-index="3">
    <strong>Has an outcome?</strong>
    <small>Move it to <code>projects/</code>.</small>
  </div>
  <div class="dz-card dz-tone-muted fragment" data-fragment-index="4">
    <strong>Reference or delete</strong>
    <small>If it is reusable, file it. Otherwise, remove it.</small>
  </div>
</div>
