From Noise to Intent

Reliability gets sharper when we stop asking only whether systems look strange and start asking whether users can still complete what matters.

Pain Revelation

Threshold-heavy alerting tells you the system is abnormal, but not whether the payment experience is actually broken for users.

What responders receive

CPU > 90% Queue depth rising Retries elevated Disk 85% Worker lag Pod restart Latency p95 drift Cache miss burst

What leadership and users need to know

Can a customer still authorize, submit, and confirm a payment?
System abnormal? Maybe
User impacted? Unknown

Intuitive Solution

Define reliability around Critical User Journeys: the steps users must complete to get real value.

Start with the journey

Model what the user is trying to accomplish, not just which components emitted metrics.

Map the required path

Include every backend dependency needed for the activity to complete successfully.

Grade what is critical

Not every interaction deserves the same SLO investment or paging urgency.

Measure the whole flow

Track the end-to-end path so failure means users cannot finish the outcome that matters.

Impact Statements From CUJs

Once the journey is explicit, incidents can be described in user terms instead of component symptoms.

Weak signal

Risk service p95 is elevated and queue depth is rising.

Technically accurate, but it does not say what users cannot do.

CUJ impact

Customers can browse and authorize, but payment confirmation is failing for checkout attempts.

The responder knows where the user journey breaks.

Incident statement

Payment completion is degraded for a critical revenue journey; page now and diagnose the confirmation path.

The business impact and response policy are visible.

CUJs turn telemetry into impact statements: who is affected, what they cannot complete, and why the response matters.

Trap of Convention

The trap is not bad observability. The trap is treating component health as a proxy for user success.

What the team optimizes for

  1. Every service gets thresholds, dashboards, and owners.
  2. Each subsystem can explain whether it looks normal.
  3. The incident room still has to infer whether the user journey is broken.

Where the inference fails

Gateway healthy
Risk service healthy
DB healthy
Confirmation step failing

The dashboard says the parts are mostly fine. The customer still cannot finish the outcome that matters.

Conviction Statement

Thresholds tell you when systems look odd. CUJ-based SLOs tell you when users are failing. You need both, with different jobs.

Threshold alerts

Early warning

Route to Slack, tickets, and prevention workflows.

CUJ SLO alerts

Paging signal

Page only when the user journey is burning fast enough to matter now.

Diagnostics

Explanation layer

Use metrics, logs, traces, and deploy history to answer why.

AI Incident Response Needs Intent

AI becomes useful when it is handed a failing journey, not just a pile of abnormal signals.

1 Impact-framed alert Payment completion is burning budget and confirmation is the failing step.
2 AI narrows the search It prioritizes deploys, logs, traces, and dependencies tied to the confirmation path.
3 Responder gets evidence The output is a ranked hypothesis with supporting signals, not a generic summary of dashboards.
4 Action gets sharper Rollback, mitigation, or owner escalation is tied to the user impact statement.

Without CUJ context, AI incident response starts from noise. With CUJ context, it investigates intent.
